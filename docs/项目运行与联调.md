## 三终端启动（仿真）

```
终端1 ── Gazebo + bridge
终端2 ── localization（EKF）
终端3 ── Nav2 + RViz
终端4 ── （可选）converter、航点、手测
```

| 步骤 | 命令 |
|:---:|------|
| **1** | `ros2 launch workspace_gz simulation.launch.py` |
| **2** | `ros2 launch workspace_ros localization.launch.py use_sim_time:=true` |
| **3** | `ros2 launch workspace_nav nav2.launch.py use_sim_time:=true`（地面站联调加 **`enable_mission_bridge:=true`**） |

地面站 / 上层任务（**推荐** `enable_mission_bridge:=true`）：

- **`enable_mission_bridge:=true`**：一并启动 `mission_bridge` + `nav_status_aggregator`；**`map_yaml_path` 由 launch 自动与 Nav2 的 `map:=` 一致**，换图只改一处（见下「Launch 参数」）。
- 提供 **Service**（`send_waypoints` / `set_pause` / `emergency_stop` / `cancel_mission`）+ **Topic 兼容**（`/waypoint` / `/color_code` / `/gcs_mission/cancel`）。
- **Service 方式**：直接获得 `success/message` 响应，无需猜测是否被接受。航点使用 map 坐标系 PoseStamped。
- **Topic 方式（兼容保留）**：写 **`waypoints.json` / `target_buoy.json`** 并逐点调用 **FollowWaypoints**。载荷字段与语义统一由 **`gps_map_conversion.parse_waypoint_message`** 解析（支持 `explicit_replan` 等；**`start_pulse` 下的 `mission_start_topic` Empty 亦视为显式启动**）。
- **航点节拍 / 抢占 / Cancel / 颜色落盘**：由 `mission_bridge.launch.py`（或 include 等价参数）控制，详见下节 **参数表**。若只希望写 `waypoints.json` 而不常驻桥接，改用 `waypoint_transform`。
- **仅落盘不写 Nav2**：`ros2 run workspace_nav waypoint_transform`（勿与 **`mission_bridge` 并行**，避免争用同一话题与 FollowWaypoints）。

##### `mission_bridge` 参数与行为（复盘与改参）

| 参数（ROS） | Launch 默认值 | 行为摘要 |
|-------------|---------------|----------|
| `waypoint_command_mode` | `debounce` | **`immediate`**：每条 `/waypoint` 即时评估。**`debounce`**：静默 `waypoint_commit_delay_sec` 后只执行最后一次航线，缓和双源交替发包。**`start_pulse`**：只缓冲，`mission_start_topic` 上 **`std_msgs/Empty` 脉冲**后才执行 |
| `waypoint_commit_delay_sec` | `0.45` | debounce 静默窗口（秒）；过小会失去防抖意义 |
| `mission_start_topic` | `""` | **`start_pulse` 必填**（例如 `/gcs_mission/start`） |
| `mission_cancel_topic` | `/gcs_mission/cancel` | 订阅 **`Empty`**：**Cancel Nav** 清空 debounce/start_pulse **缓冲**，若正在航行则 **cancel** 当前 FollowWaypoints |
| `allow_replace_running_mission` | `false` | 航行中若 `/waypoint` 仅为周期性刷新且载荷里 **`explicit_replan` 为假**，一般不抢占当前 FollowWaypoints（实船更安全）；运动中自动换路须 GCS 显式重申或设为 `true` |
| `allow_repeat_identical_route` | `false` | 同一 mission hash **完成后**是否允许立刻再跑 |
| `discard_watchdog_sec` | `4` | preempt 后置位「丢弃下一轮 goal 结果」的兜底超时（≤0 关闭） |
| `suppress_passive_waypoints_after_cancel` | `true` | **Cancel 后**：在 **`immediate` / `debounce`** 下丢弃无显式重申的被动 `/waypoint`，避免短时无速度后又自动重派。**恢复**：**Run Mission**（首帧 `explicit_replan: true`）或 **`start_pulse` + start Empty**。在 **`start_pulse`** 下仍可更新缓冲航线，但只有 **Empty** 才会启动，故不会因被动包单独「自动起航」 |
| `target_buoy_force_rewrite` | `false` | 为 `false` 时 **语义同色**不写盘（减刷屏）；为 `true` 则每次 `/color_code` 都重写 |
| `target_buoy_min_write_period_sec` | `0` | **任意颜色**写入 `target_buoy.json` 的**最短间隔**（秒）；与上一行并行，用于压住 GCS 红/绿语义**真在交替发布**时的频繁落盘。若长期只见「绿↔红」写日志，可先确认 GCS 是否在发两套语义 |
| **Service 接口（v4.0 新增）** | | |
| `mission_bridge/send_waypoints` | — | Service：接收 `PoseStamped[]` 航点 + `mission_id`，返回 `success`/`message`。IDLE/COMPLETED/FAILED 时直接执行；RUNNING/PAUSED + `allow_replace_running_mission=true` 时抢占 |
| `mission_bridge/set_pause` | — | Service：`pause=true` 暂停（保存进度→PAUSED），`pause=false` 继续（从断点→RUNNING） |
| `mission_bridge/emergency_stop` | — | Service：清空一切进入 EMERGENCY；退出需发 `/gcs_mission/cancel` Empty |

**源码与 launch**：`src/YILDIZ-USV/workspace_nav/workspace_nav/mission_bridge.py`、`workspace_nav/launch/mission_bridge.launch.py`。

##### Launch 参数（`nav2.launch.py` + mission 栈）

| Launch 参数 | 默认 | 说明 |
|-------------|------|------|
| `enable_mission_bridge` | `false` | `true` 时 include `mission_bridge.launch.py` |
| `map` | `config/map_hk.yaml` | Nav2 与 mission **共用**；子 launch 自动传入 `map_yaml_path`，**勿再单独设地图** |
| `mission_bridge_params_file` | `""`（空） | 可选 mission 栈 YAML；**不是** Nav2 的 `params_file`。为空则仅用 launch 内联默认（`debounce`、`/gcs_mission/cancel` 等） |
| `params_file` | `nav2_params.yaml` | **仅 Nav2**；勿与子 launch 同名，否则曾导致 Nav2 读到空路径报错（已改为子 launch 内 `mission_stack_params_file`） |

仿真默认里程计：`mission_bridge` 使用 **`/odometry/filtered`**（与 `nav2_params.yaml`、EKF 一致），**一般无需改 `odom_topic`**。

**为何需要地图与里程计**

- **地图 YAML**：GCS `/waypoint` 为 WGS84；`mission_bridge` 从 `map.yaml` 读 `ref_gnss*` 与 `origin`，转成 Nav2 **map 坐标** 再写 `waypoints.json` / 调 FollowWaypoints。与 Nav2 不同源会整段偏移。
- **里程计**：收到 odom 前保持 `WAITING_SYSTEM` 不发 goal；航行中用船位做航点容差判断。`nav_status_aggregator` 亦用 odom 拼 `/nav_status`。

**可选：加载 mission 栈 YAML**（改 GCS 话题、debounce、`vehicle_id` 等时）

```bash
ros2 launch workspace_nav nav2.launch.py use_sim_time:=true enable_mission_bridge:=true \
  mission_bridge_params_file:=$(ros2 pkg prefix workspace_nav)/share/workspace_nav/config/mission_stack.example.yaml
```

**单独起 mission 栈**（Nav2 已在其它终端）：

```bash
ros2 launch workspace_nav mission_bridge.launch.py use_sim_time:=true \
  map_yaml_path:=$(ros2 pkg prefix workspace_nav)/share/workspace_nav/config/map_hk.yaml
```

##### 常见问题（mission_bridge）

| 现象 | 排查方向 |
|------|-----------|
| 点 **Cancel Nav** 后不久又有速度 / 全局规划像在跑 | GCS **1 Hz `/waypoint` 不带 `explicit_replan`** → bridge 又重新 `_execute_mission`。保持 **`suppress_passive_waypoints_after_cancel:=true`**，或改正 GCS **首包显式**。若仍存在，核对是否还有**其它客户端**在向 Nav2 发路径 |
| `target_buoy.json` 红绿交替写 | 常为 GCS **真在轮换语义**（两色任务）；可先开 **`target_buoy_min_write_period_sec`** 限频，并保留「同色不写」 |


默认海图：**`workspace_nav/config/map_hk.yaml`**（与 `nav2.launch` 默认 `map:=` 一致）。换图时务必同时核对 **`workspace_ros/config/navsat.yaml`** 里的 **`datum`** 与地图 **`ref_gnss*`**（见下）。换图示例：

```bash
ros2 launch workspace_nav nav2.launch.py use_sim_time:=true \
  enable_mission_bridge:=true \
  map:=/绝对路径/你的图.yaml
```

`enable_mission_bridge:=true` 时 **不必**再为 mission 单独传 `map_yaml_path`。

#### `navsat` 参考点会自动读海图吗？

**不会。** `navsat_transform_node` 只使用 **`navsat.yaml`** 中的 **`datum: [纬度, 经度, 绕 z 航向_rad]`**，**不会**打开 `map_hk.yaml` / `map.yaml`。换图或改 **`ref_gnss*`** 后要**手改** `datum`，注意地图里是 **`[经度, 纬度]`**，`datum` 里是 **`[纬度, 经度]`**。你现在的 **`datum: [31.48618175, 120.36793468, 0.0]`** 与 **`map_hk.yaml`** 里 **`ref_gnss_10` / `ref_gnss`** 是一致的。

## Nav2 → 仿真推进器

仿真 **不要** 使用实船的 MAVROS 速度桥。使用包内 **`converter`**：

```bash
ros2 run workspace_ros converter
```

Nav2 控制器发布 **`/cmd_vel_nav`**（见 `nav2_params.yaml` 与 bringup remap）；`converter` 将其转为 Gazebo 推进器话题。键盘手控可用 **`manual_control`**（见包说明）。

### USV S 线/原地画圈：根因与调参

当船出现 **S 形横向振荡**（左右来回摆）或 **原地打转** 时，本质是 **欠阻尼横向跟踪振荡**——Nav2 控制器默认按地面小车假设追路径，USV 的惯性和滑移形成了相位延迟的反馈振荡。

更完整的 RPP 参数说明见 [`Nav2参数详解与调参指南.md`](./Nav2参数详解与调参指南.md) §2.2–2.3。

#### 前瞻距离 (lookahead) 过小 / 过大（几何原因）

RPP 在路径上取前瞻点（carrot），距离 **L** ≈ `clamp(速度 × lookahead_time, min_lookahead_dist, max_lookahead_dist)`。

| 情况 | 原因（几何） | 表现 |
|------|----------------|------|
| **L 过小** | carrot 常落在 **船最小转弯半径 \(R_{\min}\) 内侧**；船 **还没转完**，前瞻点已滑到船侧/近处，曲率指令来回翻转 | 朝目标走时 **S 弯**、左右猛修舵；严重时像 **向反方向走**（`allow_reversing: false` 时无法倒车，只能大角速度“拧”） |
| **L 过大** | carrot 在弯道前方很远，船沿切线 **直行段过长** | **有效转弯半径过大**；目标在船 **后方** 时 **先向前冲很远** 才开始拐 |

调参方向：**过小 → 增大 `min_lookahead_dist`（宜 ≥ \(R_{\min}\) 与船头前伸量）并先降速**；**过大 → 减小 `max_lookahead_dist` / `lookahead_time`**，但勿把 min 收到弯内侧。仿真默认 L 约 1–3 m，实船常 3–8 m，同一场景表现不同是正常现象。

#### 根因链

```
粗分辨率地图 (如 shanxi 7.7m/pixel)
  → SmacPlanner 路径即便经过 smoother，方向跳变大
  → RPP 检测路径曲率大 → 线速度压到 regulated_linear_scaling_min_speed (0.1 m/s)
  → converter: linear_scale×0.1 ± angular_scale×大angular → 差动推力占比过大
  → 船低速原地转向，刚对准一点路径
  → 线速度恢复 → 船冲出去 → 冲过路径
  → RPP 检测偏左 → 向右修
  → 还没修完，路径又更新了 → 来回摆 → S 线
```

三个因素互相放大：

| 层面 | 问题 | 表现 |
|------|------|------|
| **地图** | 分辨率过粗（7.7m/pixel） | 路径点空间尺度大，方向跳变，横向误差不连续 |
| **控制器** | RPP 参数偏小车 | `min_lookahead_dist: 1.0m` 对船太小，频繁追 carrot；`desired_linear_vel: 1.2` 太快 |
| **converter** | 无速度闭环，无角速度-前进耦合 | 大 angular 时 forward 不降，差动限幅不足，一侧反推致原地转 |

#### converter 改造（角速度-前进耦合）

核心原则：**转弯时必须降速**。角度误差大时降低 forward，角度误差小时恢复。

```python
forward = self.linear_scale * msg.linear.x
turn = self.angular_scale * msg.angular.z

# angular.z 越大，前进推力越小
w = abs(msg.angular.z)
if w > 0.4:
    forward *= 0.3
elif w > 0.25:
    forward *= 0.5
elif w > 0.1:
    forward *= 0.75

# 差动限幅，防止一侧反推导致原地转
max_turn = abs(forward) * 0.5
turn = max(min(turn, max_turn), -max_turn)

left_thrust = forward - turn
right_thrust = forward + turn
```

#### RPP 控制器适配船体

```yaml
# 对比：原参数（偏小车）→ 建议（船）
desired_linear_vel: 1.2 → 0.3~0.5       # 先降速，闭环调好再升
min_lookahead_dist: 1.0 → 3.0            # 船不要盯太近
max_lookahead_dist: 3.0 → 8.0            # 看远一点
lookahead_time: 2.0                       # 保持
regulated_linear_scaling_min_speed: 0.1 → 0.3  # 转弯时保持舵效
regulated_linear_scaling_min_radius: 2.0 → 1.0 # 允许更急的弯
```

#### 地图分辨率

大范围地图（如水库 7.7m/pixel）建议：
- 全局规划用粗图可以，但 **局部控制区域裁剪到 1.0~2.0 m/pixel**
- 或规划阶段降采样、控制阶段用细图
- 直接给 Nav2 用 7.7m/pixel 会导致路径点间空间尺度大，方向跳变

#### 调参顺序

1. **converter**：加角速度-前进耦合 + 差动限幅
2. **RPP**：降速 + 加大 lookahead
3. **地图**：裁剪或降分辨率
4. 每步改完在仿真中验证，S 线幅度应逐步减小

> **根本限制**：Nav2 控制器本质为地面小车设计，USV 完全消除 S 线需要船体控制层（航向误差 → 速度-转向耦合 → 左右推），而不是仅靠参数调优。但上述改动可在仿真中显著改善。

## 航点与地面站

**推荐方式（Service 接口，有 ACK）**：上层直接调 `mission_bridge` 的 ROS2 Service，无需额外启动 `waypoint_transform` 等节点。返回 `success=true/false` 明确指令是否被接受。

```bash
# 下发航线（map 坐标系 PoseStamped）
ros2 service call /mission_bridge/send_waypoints m_common/srv/SendWaypoints \
  "{waypoints: [
    {header: {frame_id: map}, pose: {position: {x: 10.0, y: 20.0}, orientation: {w: 1.0}}},
    {header: {frame_id: map}, pose: {position: {x: 30.0, y: 40.0}, orientation: {w: 1.0}}}
  ], mission_id: 'm001'}"

# 暂停 / 继续
ros2 service call /mission_bridge/set_pause m_common/srv/SetPause "{pause: true}"
ros2 service call /mission_bridge/set_pause m_common/srv/SetPause "{pause: false}"

# 急停（退出需发 cancel topic）
ros2 service call /mission_bridge/emergency_stop m_common/srv/EmergencyStop

# 退出急停
ros2 topic pub /gcs_mission/cancel std_msgs/msg/Empty "{}" --once
```

**常用操作流程**：

```
# 1. 启动前确认状态
ros2 topic echo /mission_bridge/state --once    # 应输出 data: IDLE

# 2. 下发航线 → 立即执行
ros2 service call /mission_bridge/send_waypoints ...   # Response: success=True

# 3. 运行中暂停
ros2 service call /mission_bridge/set_pause "{pause: true}"   # → PAUSED

# 4. 从断点继续
ros2 service call /mission_bridge/set_pause "{pause: false}"  # → RUNNING

# 5. 取消任务
ros2 topic pub /gcs_mission/cancel std_msgs/msg/Empty "{}" --once   # → IDLE
```

**兼容方式（Topic，保留兼容）**：与旧流程一致。

```bash
ros2 run workspace_nav waypoint_transform
ros2 run workspace_nav waypoint_with_state
```

分立节点与 **`mission_bridge` 勿重复运行**（会争用同一 `/waypoint` 与 FollowWaypoints）。

- **`waypoint_transform`**：单次任务写 **`waypoints.json`**；默认 **`datum_source:=map_yaml`**（默认同包 **`map_hk.yaml`** 的 **`ref_gnss*`**）；`datum_source:=first_gps` 时用仿真 GPS（默认 **`/roboboat/sensors/gps/navsat`**）。
- **`waypoint_with_state`**：默认 **`/odometry/filtered`**，从 **`waypoints.json`** 跟进 Nav2。

### Service vs Topic 对比

| | Service（推荐） | Topic（兼容） |
|------|-----------|--------|
| 下发航线 | `/mission_bridge/send_waypoints` | `/waypoint` (JSON) |
| 暂停/继续 | `/mission_bridge/set_pause` | 无 |
| 急停 | `/mission_bridge/emergency_stop` | 无 |
| ACK | **有**（success/message） | 无（需看 /nav_status 间接确认） |
| 坐标 | map 坐标系 PoseStamped | WGS84 经纬度 JSON |
| 启动条件 | IDLE/COMPLETED/FAILED | 同上 + 需了解 waypoint_command_mode |
