# 实船调试（MAVROS + PX4）

本文档说明在 **真机 / 实船** 上通过 **MAVROS** 连接 **Pixhawk（运行 PX4 固件）** 时的常用启动方式与自检要点。**与仿真并列的一页总启动表**（含终端顺序与模式 B/A 命令）见仓库根目录 **[`docs/项目运行与联调.md`](../../../docs/项目运行与联调.md)**。**TF 树、各段发布者与调试命令（复盘）**见本文 **「TF 坐标系总览与复盘调试」**。更完整的 **仿真→真船迁移分层清单** 与 **MAVROS 话题语义表**，见 **[`实船迁移与MAVROS话题.md`](./实船迁移与MAVROS话题.md)**。**按文件逐项改哪些参数/话题**，见 **[`实船配置修改清单.md`](./实船配置修改清单.md)**。

仿真（Gazebo）流程仍以 `README.md`、`PROJECT_ARCHITECTURE_AND_NAV2.md` §2 为准。

## 前提

- **操作系统**：Ubuntu 22.04（与仓库 README 一致）
- **ROS 2**：Humble
- **飞控固件**：本仓库实船场景按 **PX4** 说明；常见硬件如 `Auterion PX4 FMU v6C` 等通过 USB 连接后，在主机上通常表现为 `/dev/ttyACM0`（以 `ls /dev/ttyACM*` 为准）。
- **权限**：用户需在 `dialout` 组，否则无法打开串口设备。

已安装系统包示例：

```bash
sudo apt install ros-humble-mavros ros-humble-mavlink
```

## MAVROS 启动（PX4）

每个终端先加载 ROS 环境：

```bash
source /opt/ros/humble/setup.bash
```

**使用 PX4 插件与配置启动 MAVROS**（勿对 PX4 飞控使用 `apm.launch`）：

```bash
ros2 launch mavros px4.launch fcu_url:=/dev/ttyACM0:57600
```

### 推荐（本仓库）：叠加 USV/Nav2 用 TF 覆盖

默认官方 **`px4_config.yaml`** 里 **`local_position.tf.send` 为 `false`**，且 **`tf.frame_id` 常为 `map`**，与 **`map`→`odom`→`base_link`** 的 Nav2 链不易衔接。本包提供 **在官方配置之后加载的覆盖文件** 与独立 launch：

```bash
ros2 launch workspace_ros mavros_px4_usv.launch.py fcu_url:=/dev/ttyACM0:57600
```

- **覆盖层**：`workspace_ros/config/mavros_px4_overrides_usv.yaml`（打开 **`odom`→`base_link`** TF、关 **`global_position`** 发 TF 以免重复）。  
- **官方层**（默认路径，随安装）：`$(ros2 pkg prefix mavros)/share/mavros/launch/px4_pluginlists.yaml` 与 **`px4_config.yaml`**。  
- 若你 **换发行版/改路径**，可传参：`pluginlists_yaml:=...` **`base_config_yaml:=...`** **`override_config_yaml:=...`**。

**为何只起 MAVROS 时看不到 `map→odom`？**  
`mavros_px4_usv.launch.py` **只启动飞控桥**，不包含 Nav2 **TF** 装订。 **`map→odom`** 在本仓库由 **`real_boat_bringup.launch.py`** 拉起：默认走 **`localization.launch.py`**（末尾有 `map_to_odom_tf`）；若 **`localization_backend:=mavros_odom`** 则 **`real_boat_mavros_tf.launch.py`**：默认 **`gnss_odom_map_tf`**（从 **`map_config_yaml`** 读 **`map_origin_ref_key`** 对应角点经纬度装订；`use_gnss_map_odom_tf:=false` 时为带 **`map_odom_yaw_deg`** 的可旋转静态 `map→odom`；薄封装旧名 **`real_boat_tf_static.launch.py`**）。**仅测 MAVROS、未起 bringup** 时，`view_frames` 里缺 **`map→odom` 是正常的**；可临时：`ros2 run tf2_ros static_transform_publisher 0 0 0 0 0 0 map odom`。

**PX4 「HOME / 局域原点」不是 MAVROS YAML 里的键**  
飞机/船的 **家园点、局部NED原点**由 **PX4 固件状态与地面站约定**维护，一般不写进 **`mavros_px4_overrides_usv.yaml`**。常见设定方式：**QGroundControl「设 HOME」**、首次 **GPS** 就绪后 PX4 **自动 HOME**（与 **`COM_HOME_*`**、`EKF2_*` 等固件参数相关）、或通过 **MAVLink** **`MAV_CMD_DO_SET_HOME`**（MAVROS 侧可查 **`cmd/command`**、`home_position` 等相关服务/话题——以本机 **`ros2 interface list`**、PX4 + mavros 版本为准）。与你的 **栅格 `map`** 对齐仍属工程约定：**实船默认 `map_real_boat_hk.yaml`、仿真 `map.yaml`** 中 **`ref_gnss*`**、`navsat`/datum、`HOME` 一致；见下文「map/odom/HOME」及 **「用 ROS 2 服务设置 PX4 HOME」**（含可复制命令）。

仍可直接用上游 **`px4.launch`**；此时需**自行**在官方 `px4_config.yaml` 里改 **`local_position`** / TF，或接受无 **`odom`→`base_link`** 时另起 **EKF 仅发 TF**。

说明：

- `fcu_url` 中 **设备路径** 按本机实际修改（如 `/dev/ttyACM1`）。
- USB 连接飞控时，串口 **波特率** 在部分驱动上可能不起作用，但保持 `57600` 或按团队约定填写即可；若官方文档对 USB 有指定写法，以固件/地面站文档为准。

### 可选：转发到地面站

若需 MAVROS 与 QGroundControl 等同时存在，由 MAVROS 负责转发时，可设置 `gcs_url`（具体格式见 `mavros` 官方说明）。单机直连调试可暂留默认。

## 链路自检（ROS）

MAVROS 起来后：

```bash
source /opt/ros/humble/setup.bash
ros2 topic echo /mavros/state
```

期望 **`connected: true`**。若长时间为 `false`，检查端口占用、线缆、`fcu_url`、以及飞控是否正常上电。

**里程话题（任一暂行方案都会用到自检）**：

```bash
ros2 topic echo /mavros/local_position/odom --once
ros2 topic hz /mavros/local_position/odom
```

若频率为 **0**：多为 **无解算**（GPS 不稳、 indoor 室内无定位）、或飞控未完成 **磁力计/罗盘** / **校准**、`EKF_*` / `SYS*` 等与估计算法相关的检查未通过。应在 **地面站 / `statustext`** 查清原因再继续。

**TF（尤其使用下面「PX4 融合里程」模式时）**：

```bash
ros2 run tf2_tools view_frames
```

核对是否存在连贯的 **`map` → `odom` → `base_link`**（或与 Nav2 **`robot_base_frame`** 一致的中间名）。如出现 **重复的 `odom`→车体** broadcast，或未连上 `map`，Nav2 会报警或无法控制。

### `map_ned` / `odom_ned` / `*_ned` 与「谁来发 ROS 这边的 TF」（模式 B）

飞控一侧惯用 **NED**；部分 **MAVROS/PX4 配置**会把 TF 打成 **`map_ned` ↔ `odom_ned` ↔ `base_link`（或 body 帧带 `_ned`）** 的**另一棵树**，和你们 Nav2、`real_boat_mavros_tf.launch.py`（旧名 **`real_boat_tf_static.launch.py`**）里用的 **`map` / `odom` / `base_link`** **名字不一致**，所以 **`map`→`odom`** 发布后， **`odom` 下面仍然没有 `base_link`**——因为机体关节在 **`odom_ned`→`xxx_ned`** 上。

| 谁在仿真里干过 `odom`→`base_link` | 说明 |
|----------------------------------|------|
| **`ekf_node`（robot_localization）** | **`localization.launch.py`（模式 A）** 里 **`publish_tf: true`**，EKF **广播 `odom`→`base_link`**。**模式 B（`mavros_odom`）故意不启动 EKF**，这条链要自己接。 |

**推荐做法（按优先级）**

1. **让 MAVROS 直接发 ROS 命名的 TF**：在 MAVROS **随 `px4.launch` 加载的插件 YAML**（本机通常在 **`/opt/ros/<distro>/share/mavros/...`**；或你们自复制的参数文件）里，找到 **`local_position` / TF 相关项**，使 **局域位姿插件**向外发布 **`odom`→`base_link`（或与 Nav2 一致的 `robot_base_frame`）**，并统一到 **ROS/ENU** 语义（不要一套 `_ned`、一套无后缀混用——具体键名依 **mavros 版本**，用 **`ros2 param list`** / 官方 **`px4_config.yaml`** 对照）。必要时 **关掉**重复的 **`*_ned`** 广播，防止 **两组 `tf` 抢同一父子关系**。  

2. **仍保留 `_ned` 树时**：可另加 **`static_transform_publisher`** 或 **`tf2_ros`** 中继，把 **`odom`↔`odom_ned`**（常为固定旋转）接起来，再让你们现有的 **`map`→`odom`** 接上；这要算清 **NED↔ENU 旋转**，不推荐长期硬扛，除非你团队已书面约定。  

3. **再回到 `robot_localization`「只搭桥」**：订阅 **`/mavros/local_position/odom`**，**关掉 MAVROS 发 `odom`→车体的 TF**（避免两套），仅用 **`ekf_node`** 且 **`publish_tf: true`**、**输出帧名为 `odom` / `base_link`**——相当于用 EKF **做坐标系更名/滤波**，数据源仍是 MAVROS 里程。**不要**与本仓库 **「模式 A」全功能 EKF+navsat GPS 输入**在未理清参数前叠在一起，以免双重融合。

自检：改完后 **`ros2 run tf2_tools view_frames`** 应能看见 **单一路径**上到 **`base_link`**，且 **`/mavros/local_position/odom` 消息里的 `header.frame_id` / `child_frame_id`** 应与 TF 父子名一致或可经一层静态变换解释。

**优先**：已用本仓库 **`ros2 launch workspace_ros mavros_px4_usv.launch.py`**（覆盖 **`mavros_px4_overrides_usv.yaml`**）时，一般应出现 **`odom`→`base_link`**；若仍只有 **`*_ned`**，再对照飞控/MAVROS 版本与 **`ros2 param get /mavros/local_position`**。

## TF 坐标系总览与复盘调试

汇总 **谁发布哪一段 TF**、**仿真 / 模式 A / 模式 B 差异**与 **联调命令**；与上节「`*_ned`」互补。架构侧见 [`PROJECT_ARCHITECTURE_AND_NAV2.md`](./PROJECT_ARCHITECTURE_AND_NAV2.md) §4.0。

### 目标拓扑（Nav2 约定）

Nav2 使用 **`global_frame: map`**、**`robot_base_frame: base_link`**（见 `nav2_params*.yaml`）。目标是从 **`map`** 能变换到 **`LaserScan.header.frame_id`** 等传感器帧。本仓库典型结构：

```text
map
 └── odom                          ← localization 恒等静态，或 real_boat_mavros_tf（默认 gnss_odom_map_tf / 可选恒等）
      └── base_link                ← 模式 B：MAVROS + mavros_px4_overrides_usv
           ├── lidar_link          ← 模式 B：static_transform_real_boat.yaml（父为 base_link）
           ├── imu_link
           ├── gps_link
           └── camera_link         （按实装保留）
```

**模式 A** 下 **`odom`→`base_link`** 由 **`ekf_node`** 发布；仿真时 **`robot_state_publisher`（URDF）** 与 **`static_transform.yaml`** 共同承担 Gazebo 侧传感器命名。

### 发布者对照表

| TF 段 | 模式 B（`mavros_odom`） | 模式 A（`robot_localization`） | 仿真（README 四步） |
|-------|------------------------|--------------------------------|---------------------|
| **`map`→`odom`** | **`real_boat_mavros_tf`** 内 **`gnss_odom_map_tf`**（默认）或 `tf2_ros/static_transform_publisher` | **`localization.launch`** 末尾同名节点 | **`localization.launch`** |
| **`odom`→`base_link`** | **MAVROS** + **`mavros_px4_overrides_usv.yaml`** | **`ekf_node`** | **`ekf_node`** |
| **传感器相对船体** | **`static_transform_real_boat.yaml`**（默认；可 **`static_transform_file:=...`**） | **`static_transform.yaml`**（与 Gazebo frame 链配套） | URDF 关节 + 同上 YAML |

### 仅 MAVROS、未起 bringup

一般 **没有** **`map`→`odom`**；可临时：

```bash
ros2 run tf2_ros static_transform_publisher 0 0 0 0 0 0 map odom
```

### 激光 / 点云 `frame_id`（避障）

Nav2 要把障碍画进 **`odom`/`map`** 下的代价地图，**传感器消息的 `frame_id` 必须落在连贯 TF 树上**（经 **`base_link`** 连到 **`odom`/`map`**）。

- **`sensor_msgs/LaserScan`**（例如经 **`mavros_roboboat_relay`** 映到 **`/roboboat/sensors/lidar/scan`**）：若 **`header.frame_id`** 不是 **`lidar_link`**，在 **`static_transform_real_boat.yaml`**（或副本）中增加 **`base_link`→`<该 frame>`**，或改驱动配置。
- **`sensor_msgs/PointCloud2`**（实船 **`nav2_params_real_mavros.yaml`** 默认 **Livox MID-360** 等）：核对 **`header.frame_id`**（常见为 Livox 驱动自带帧名，未必等于 **`lidar_link`**），同样用 **静态 TF** 或驱动参数接到 **`base_link`**。

```bash
# 若仍走二维扫（relay）
ros2 topic echo /roboboat/sensors/lidar/scan --once
# Livox 点云（话题以驱动为准；默认配置为 /livox/lidar）
ros2 topic echo /livox/lidar --once
ros2 run tf2_ros tf2_echo base_link <点云 frame_id>
```

### 调试命令清单

```bash
ros2 run tf2_tools view_frames
ros2 run tf2_ros tf2_echo map base_link
ros2 param get /mavros/local_position tf.frame_id
ros2 param get /mavros/local_position tf.child_frame_id

```

### 常见问题速查

| 现象 | 排查方向 |
|------|----------|
| **`base_link` 与各 `*_link` 断开** | 模式 B 是否启用 **`static_transform_real_boat.yaml`**（勿套用仿真 **`static_transform.yaml`** 里「`*_link`→`roboboat/...`」拓扑到 MAVROS **`base_link` 链**）；**`static_transform_file`** 路径是否正确。 |
| **`map`→`odom` 缺失** | 是否已 **`real_boat_bringup`** 或 **`localization.launch`**。 |
| **`odom`→`base_link` 缺失** | **`mavros_px4_usv.launch.py`** 与 **`mavros_px4_overrides_usv.yaml`**；飞控 **`/mavros/local_position/odom`** 是否有输出。 |
| **模式 A 下 TF 冲突** | 勿让 MAVROS **`local_position`** 与 **`ekf_node`** **重复广播** **`odom`→`base_link`**。 |

## 定位后端二选一（实船）

| 模式 | `real_boat_bringup.launch.py` | Nav2 | 备注 |
|------|------------------------------|------|------|
| **A：本仓库 EKF（与仿真一致）** | `localization_backend:=robot_localization`（默认） | **`nav2.launch.py`** + **`use_sim_time:=false`**（默认仍用 **`nav2_params.yaml`** → **`/odometry/filtered`**） | `/odometry/filtered` **不要**再与 MAVROS **`odom` TF 重复发给同一子坐标系 |
| **B：暂行 PX4 / MAVROS 融合里程（推荐先试通）** | `localization_backend:=mavros_odom` | **`nav2_real_mavros.launch.py`**（默认 **`nav2_params_real_mavros.yaml`** → **`/mavros/local_position/odom`**） | 也可用 **`nav2.launch.py use_mavros_odometry:=true`**（运行时 merge **`nav2_mavros_odom_overlay.yaml`**）。**不起**包内 **`ekf`** / **`navsat_transform`** |

**GNSS 与局域里程**：`/mavros/local_position/odom` 通常**已含**飞控对 **GNSS+IMU** 的融合；另需监视原始经纬度可看 **`/mavros/global_position/raw/fix`**。Nav2 要的 **`odom`** 用前者即可；**栅格 `map` 对齐**仍依赖 **`map.yaml`/datum** 与 **PX4 HOME** 的工程约定，否则地图上整体会偏。

**模式 B：推荐启动示例**（新开终端，已 `source install/setup.bash`）。**Nav2 与 bringup 须共用同一份地图 YAML**（`map_server` 与 **`gnss_odom_map_tf`** 解析锚点一致），否则 **`map→odom` 平移会错**、船会整片偏出栅格。

```bash
# 0) MAVROS（推荐本仓库 PX4 TF 覆盖）
ros2 launch workspace_ros mavros_px4_usv.launch.py fcu_url:=/dev/ttyACM0:57600

# 1) 传感器桥 + TF（默认 gnss_odom_map_tf + static_transform_real_boat.yaml），不跑 EKF
#    默认 map_config_yaml = 包内 map_real_boat_hk.yaml；换图时与 nav2 的 map:= 传同一路径。
ros2 launch workspace_ros real_boat_bringup.launch.py \
  use_sim_time:=false \
  localization_backend:=mavros_odom \
  scan_src:=/你的激光话题 \
  map_config_yaml:=/path/to/same_as_nav2_map.yaml \
  map_origin_ref_key:=ref_gnss_10

# （可选）锚点对应 map 原点 (0,0) 的另一角点时，例如 ref_gnss_00；固定栅格 vs ENU 偏角：`map_odom_yaw_deg:=90.0`。

# 2) Nav2 — 实船专用 launch，必须与上一步 YAML 对齐
ros2 launch workspace_nav nav2_real_mavros.launch.py map:=/path/to/same_as_nav2_map.yaml
```

换图 **`map.yaml`** 而未改 **`map_config_yaml`** 是导致「轴向对但整条船偏移」的常见原因。**默认不写 `map_config_yaml`** 时已与 **`nav2_real_mavros` 默认图**同源（均指向安装后的 **`workspace_nav/share/.../map_real_boat_hk.yaml`**）。

**避障传感器（模式 B）**：实船默认参数 **`config/nav2_params_real_mavros.yaml`** 中，**`local_costmap`** 使用 **`nav2_costmap_2d::VoxelLayer`** + `inflation_layer`，观测 **`sensor_msgs/PointCloud2`**，默认话题 **`/livox/lidar`**（`livox_cloud` 源）。**`global_costmap`** 使用 **`static_layer` + `voxel_layer` + `inflation_layer`**（远距离预判，max 40m）。注意非滚动窗口下动态物离开后标记可能残留（ghost）；水面动态物少时影响不大，岸测有大量 ghost 时可临时 `enabled: false`。**仿真用** **`nav2_params.yaml`** 仍为 **`ObstacleLayer` + `/roboboat/sensors/lidar/scan`**，二者互不覆盖。

**若 Livox 话题不是 `/livox/lidar`**：只改该文件中 **`voxel_layer.livox_cloud.topic`**（当前仅在 local_costmap 中有一处）。

**若仍使用二维激光**：把 local costmap 的 **`plugins`** 改回 **`obstacle_layer`** + **`inflation_layer`**，并恢复 **`obstacle_layer`** + **`LaserScan`** 配置；或通过 **`scan_src:=...`** 经 **`mavros_roboboat_relay`** 转发到 **`/roboboat/sensors/lidar/scan`** 且 Nav2 仍配置为 **`ObstacleLayer`**（需与当前仓库默认 yaml 二选一，勿混用）。

**临时关闭局部障碍检测（纯测试导航/海图对齐时）**：将 `local_costmap` 的 `voxel_layer.enabled` 改回 `false` 即可（改完重启 Nav2），这样只靠 static_layer + inflation 走固定地图。测试完毕改回 `true`。

**注意事项（VoxelLayer + 点云）**：

- **footprint**：船体 1.6×0.8 m（保守），base_link 在船尾往前 0.2 m（非对称多边形 `[[1.4,0.4],[1.4,-0.4],[-0.2,-0.4],[-0.2,0.4]]`）。参数内 `——` 标记行均有中文注释。
- **膨胀层未改**：仍是 2D **`InflationLayer`**；VoxelLayer 在 3D 体素上标记/清除后 **投影为 2D costmap**，规划语义与原先「激光障碍层 + 膨胀」一致。膨胀 local 1.2 m / global 1.8 m。
- **必须标定高度与范围**：**`origin_z`、`z_resolution`、`z_voxels`、`min_obstacle_height` / `max_obstacle_height`**、**`obstacle_max_range` / `raytrace_max_range`** 与安装高度、水面杂波相关；不当会导致 **假障碍、漏检或 CPU 过高**。
- **mark_threshold（标记阈值）**：当前设为 **1**（需同一体素柱连续 2 帧被击中才标障碍），可滤除偶发噪点。水面杂波多时可增大到 2；需要高灵敏时改回 0（首次点击即标记）。
- **水面反射 / 动态障碍**：必要时在驱动侧或增加滤波，并复查 **`mark_threshold`** 等 Voxel 参数（见 [Nav2 Voxel Layer](https://docs.nav2.org/configuration/packages/costmap-plugins/voxel.html)）。

**时间**：模式 B 下默认参数文件已 **`use_sim_time: false`**；回放 bag 需设 **`use_sim_time:=true`** 时再显式传入。

## `gnss_odom_map_tf` 与地图锚点（模式 B，`map→odom`）

**节点**：**`workspace_ros/scripts/gnss_odom_map_tf.py`**（由 **`real_boat_mavros_tf.launch.py`** 在 **`use_gnss_map_odom_tf:=true`**（默认）时启动）。

**行为概要**：用 **`/mavros/global_position/global`**（或参数改话题）与 **`/mavros/local_position/odom`** 首帧（或持续，见 **`initialize_once`**）计算 **`map→odom`** 平移；ENU 米制与可选 **`map_odom_yaw_deg`** 一同旋入 Nav2 **`map`** 轴（详见节点头注释）。

**锚点来源（与 launch 硬编码经纬度解耦）**：

| 参数 | 说明 |
|------|------|
| **`map_config_yaml`** | 与 **`map_server` / `nav2_real_mavros` 的 `map:=`** **同一份** YAML 的绝对路径或已安装 share 路径。默认：与 **`nav2_real_mavros`** 相同的 **`map_real_boat_hk.yaml`**。 |
| **`map_origin_ref_key`** | YAML 中 **`ref_gnss*`** 键名，该项为 **`[longitude, latitude]`（度）**，须表示 **Nav2 栅格世界原点 (0,0)** 在真实地球表面上的那一角；默认 **`ref_gnss_10`**（与 **`waypoint_transform`** 等约定一致；若制图原点在另一角，改为 **`ref_gnss_00`** 等）。 |
| **`map_odom_yaw_deg`** | 栅格 **x/y** 与局地 ENU 存在 **固定绕 z 偏角**（常见 ±90°）时调节；默认 `0`。 |
| （高级）**`map_origin_latitude` / `map_origin_longitude`** | 仅当 **`map_config_yaml` 留空** 时用手工锚点；否则由 YAML 解析覆盖。 |

**算法与实现要点（复盘／改代码时从这里对）**

实现文件：**[`workspace_ros/scripts/gnss_odom_map_tf.py`](../workspace_ros/scripts/gnss_odom_map_tf.py)**。

1. **锚点**：启动时 **`lat_lon_from_map_yaml(map_config_yaml, map_origin_ref_key)`** 读出 **`[lon,lat]`**，得到 **`(lat₀, lon₀)`**。必须与 **Nav2 栅格原点 (0,0)** 在真实地球上的角点一致。  
2. **输入**（可调参数名见源码）：订阅 **`sensor_msgs/NavSatFix`**（默认 **`/mavros/global_position/global`**）与 **`nav_msgs/Odometry`**（默认 **`/mavros/local_position/odom`**）；均为 **BEST_EFFORT QoS**，与 MAVROS 常见传感器端一致。  
3. **平面近似（ENU，米）**：对当前 GNSS **`(lat, lon)`**，相对锚点 **`(lat₀, lon₀)`** 计算东向 **`east`**、北向 **`north`**（WGS84 赤道半径 **`WGS84_A = 6378137`**，纬度用 **`cos(lat₀)`** 缩放经差）。**不做 NED⇔ENU**：直接用话题里的 **odom 平移 **`x`,`y`**。  
4. **`map→odom` 平移（推导要点）**：在「ENU 对齐的平面」里有 **`Δx₀ = east − pox`**、**`Δy₀ = north − poy`**，其中 **`(pox, poy)`** 取自 **`Odometry.pose.pose.position.x/y`**。再令 **`ψ = map_odom_yaw_deg`（弧度）**：  
   - **`rx = cos(ψ)·Δx₀ − sin(ψ)·Δy₀`**，**`ry = sin(ψ)·Δx₀ + cos(ψ)·Δy₀`** → 填入 **`TransformStamped.transform.translation`**；  
   - 旋转四元数为 **绕 `map` 的 +z** 转角 **`ψ`**（**平移与姿态使用同一 ψ**，勿只旋其一）。**`translation.z`** 强制 **0**（海面 Nav2 平面约定）。  
5. **不包含**：**栅格 `resolution`/仿射缩放**、`map.yaml` 里 **`origin` 第三分量 yaw** 的自动耦合（如需与 **`waypoint_transform`** 完全一致，应由团队统一 **`map_origin_ref_key`、`map_odom_yaw_deg`** 与制图约定）；亦不融合 **高程**。若 **BBox 米制宽高** 与 **`width×resolution`** 不一致，仅靠本节点无法修正比例误差，需从源头改图或扩展算法。  
6. **时间与锁定**：可选 **`initialize_once`**（默认真）仅在首对合格 GNSS+odom 上计算后置锁，并按 **`republish_hz`** **只更新时间戳重复发布** **`map→odom`**，减轻 GPS 抖动；跟 drift 时改 **`initialize_once:=false`**。可选 **`max_data_age_sec`** 做消息_stamp 新鲜度过滤（默认 0＝不关）。  

改逻辑时：**先对 `_try_publish()` 数学**，再核对 **YAML 锚点是否与 `map_server` 同源**、`ref` **键是否与 (0,0) 一致**。

**自检**：栅格元数据在 **`/map`**（`nav_msgs/OccupancyGrid`）的 **`info.resolution` / `width` / `height` / `info.origin`**；可用 `ros2 topic echo /map --once` 对照 YAML 与现场 **`ref_gnss*`** 四至是否米制一致（分辨率与裁切不一致时会出现「轴向对、整船偏在网格外」）。

## `map` / `odom` / PX4 HOME 与海图对齐（模式 B：仅用 MAVROS、不跑 navsat_transform）

### 为何不会「自动落在地图正确位置」

- **原工程（模式 A）**：**`navsat_transform`** 按 **`navsat.yaml` 的 `datum`** 与 **`map.yaml` 的 `ref_gnss*`**，把「参考经纬度 ↔ 制图平面坐标」绑在同一套约定上；再经 **`ekf_node`→`/odometry/filtered`**，**`odom`→`base_link`** 与 **栅格 `map`** 在设计上可对齐。  
- **模式 B（不启 `navsat_transform`）**：Nav2 **`nav2_real_mavros`** 默认用 **`map_real_boat_hk.yaml` + `hk_map.pgm`**（园区 HK）；**`/mavros/local_position/odom`** 仍是 **相对 PX4 HOME**，**不会**因换图就自动与图上某点对齐。  
- **`real_boat_mavros_tf.launch.py`**（旧名 **`real_boat_tf_static.launch.py`**）中 **`map`→`odom`**：**默认 `gnss_odom_map_tf`** 用 **与 `map_server` 同源 YAML** 的 **`map_origin_ref_key`** 作锚点 + 首帧 GNSS+局域 odom 对齐（可 `initialize_once` / `republish_hz`）。**`use_gnss_map_odom_tf:=false`** 时改用 **静态 `tf2_ros/static_transform_publisher`**（平移恒为 0，可按 **`map_odom_yaw_deg`** 设绕 z 姿态）。**图上不偏**仍要求：**YAML 锚点 = 栅格 (0,0) 对应的真实角**、与 **PX4 HOME** 及 **`local_position/odom`** 约定一致，否则整体平移/转偏。

**结论**：**可以只用 MAVROS 融合位姿做导航，但不会自动完成 `navsat_transform` 那张「GNSS ↔ `map`」对齐；要么把 PX4 HOME 对齐到地图锚点，要么显式补一层 `map`↔`odom`（见下表）。**

### 怎样让船在地图上「看起来对」

| 做法 | 说明 |
|------|------|
| **对齐 HOME 到地图锚点（常用）** | **QGroundControl** 将 **HOME** 设为与 Nav2 **当前载入的地图 YAML** 中 **`ref_gnss*`** 某一角点一致（HK 实船默认 **`map_real_boat_hk.yaml`**；仿真用 **`map.yaml`**）。**或在机上用 ROS 2 调 `/mavros/cmd/set_home`**（见下节 **「用 ROS 2 服务设置 PX4 HOME」**）。须与 **`gnss_odom_map_tf` 的 `map_origin_ref_key`**、**`waypoint_transform` 的 `map_datum_ref_key`** **同一角点约定**；制图 **(0,0)** 若绑 **`ref_gnss_00`** 而 launch 仍默认 **`ref_gnss_10`**，会整图平移。设完可 **关掉 QGC**，以免与 MAVROS **争 USB 串口**。 |
| **HOME 与制图角点不一致** | 需要 **非恒等 `map`→`odom`**：**AMCL**、一次性标定、**`navsat_transform` 仅算偏置/TF（注意勿与 MAVROS **重复广播**同一 **`odom`→车体**）、或地面站/任务层 **固定偏置**。 |
| **对照 GNSS** | **`/mavros/global_position/raw/fix`** 与 **`map` 参考角点**比对；**`local_position/odom` 已含融合，但对齐 `map` 是原点约定问题，不是多订一个话题就解决**。 |

### 用 ROS 2 服务设置 PX4 HOME（对齐 `ref_gnss*`）

**适用**：模式 B、**恒等 `map→odom`**，希望 **`/mavros/local_position/odom` 原点**与**当前地图上选定的 GNSS 角点**一致（与 **QGC 手动设 HOME** 等价另一类入口）。

**服务**（以本机为准；可先运行 `ros2 service list`，再查找含 `set_home` 的服务名）：

- **`/mavros/cmd/set_home`**，类型 **`mavros_msgs/srv/CommandHome`**。

**`CommandHome.request.current_gps`（极易搞反）**  
对应 MAVLink **`MAV_CMD_DO_SET_HOME`** 的 **param1（Use Current）**（见 [MAVLink 文档](https://mavlink.io/en/messages/common.html#MAV_CMD_DO_SET_HOME)）；MavROS 实现为：**`current_gps: true` → param1 = 1 → 用飞行器当前估计位置设 HOME**，此时**不会**采用你在请求里填的经纬度；**`current_gps: false` → param1 = 0 → 用下面填写的经纬度高程**。

因此要把 HOME 定在地图 YAML 里的角点时，必须使用 **`current_gps: false`**。

**YAML 中与角点一致性**：`workspace_nav/config/map*.yaml` 内 **`ref_gnss*`** 为 **`[longitude, latitude]`（度）**；填入服务时注意 **纬度 / 经度字段不要写反**。行内 YAML 关键字后**须有空格**（错误示例：`longitude:120.37` → 可能被解析失败）。

**示例**：经纬度**必须**从你**当前** `map_config_yaml` / `map:=` 载入的那份文件里**照抄**所选 **`ref_gnss_*`** 的 **`[lon, lat]`**，勿混用旧注释或另一张图。下面仅为**格式**示意（数值以你机载 YAML 为准）：

```bash
# 任选：与团队选定的 ref_gnss_* 角点一致；此处演示 ref_gnss_00
ros2 service call /mavros/cmd/set_home mavros_msgs/srv/CommandHome "
current_gps: false
yaw: 0.0
latitude: 31.48551322
longitude: 120.36783085
altitude: 12.85
"
```

期望响应 **`success=True, result=0`**（PX4 **`MAV_RESULT_ACCEPTED`**）。再核对：

```bash
ros2 topic echo /mavros/home_position/home --once
```

- **`geo.latitude` / `geo.longitude`** 应与所选角点在 **\(10^{-6}\)～\(10^{-7}\)°** 量级一致即算对上。  
- **`position.x` / `y` 等大偏移（米）** 属正常：**HOME 已在角点，机体仍在水面别处**，局域描述的是**船相对 HOME** 的偏移。  
- **`geo.altitude`** 可能与命令里的 **`altitude` 不完全相同**（**MSL / 椭球 / 水准面** 与 MavROS 对 **`HOME_POSITION` 的换算**）；一般**以水平对齐为主**，需要垂直精度时再与队里高程基准统一。

**与 `/mavros/global_position/global` 的关系**：船未远离 HOME 时，全局经纬应与 **`home_position/home`** 接近（米级漂移常见）。

**仅读话题**：**`/mavros/home_position/home`** 一般由飞控回填，不要用 `echo` 当「设置接口」。

**高级**：存在话题 **`/mavros/home_position/set`**（向 MavROS **发布 `mavros_msgs/msg/HomePosition`**，内部发 **`SET_HOME_POSITION`**）；需自拟局部位姿等字段，不如 **`cmd/set_home`** 简单。**首帧 GNSS**：上电 HOME 常为 **GPS 就绪后 PX4 自动写入**或与 **解锁/指挥官逻辑**有关，不一定等于「串口第一个 NMEA」；对齐海图仍以 **手动设 HOME** 或上式 **显式经纬** 为准。

### 现场自检

1. RViz **Fixed Frame = `map`**。  
2. 对照 **`raw/fix`** 与 **当前地图 YAML**（默认 **`map_real_boat_hk.yaml`**）里的 **`ref_gnss*`**。  
3. **稳定固定偏差** → 多为 **HOME ≠ 制图参考点**；优先改 HOME，再考虑 **`map`→`odom` 补偿**。  
4. 需要 **贴栅格海图定点航行** 前，须完成上表策略之一；仅「能走、能避障」联调可暂时接受误差。更系统叙述见 **[`实船迁移与MAVROS话题.md`](./实船迁移与MAVROS话题.md)**「地图与 MAVROS 局域原点」专节。

## 实船：Nav2 `/cmd_vel` → PX4 OFFBOARD（推荐）

**控制链**：

```text
Nav2（controller → velocity_smoother）
  ↓ /cmd_vel   geometry_msgs/Twist
nav2_cmd_vel_to_mavros
  ↓ /mavros/setpoint_velocity/cmd_vel_unstamped
MAVROS → PX4 OFFBOARD → 混控 → 推进器
```

**差速 / 双侧推进**：ROS 侧只发 **`linear.x`** 与 **`angular.z`**；具体推力分配由 **PX4 机架与 mixer** 完成。

**Nav2 仍可能短暂输出负 `linear.x` 或低速大角速度**；本船若 **不能倒车 / 不宜原地转向**，在桥接节点用 **`forbid_reverse`**、**`min_surge_for_turn`**（与 **`nav2_params_real_mavros.yaml`** 一致）。

### MAVROS 侧观察（调试）

台架或下水联调时，建议开若干终端对照：

```bash
# 1）Nav2 或手动的目标速度（桥接节点输入）
ros2 topic echo /cmd_vel

# 2）桥接节点写入 MAVROS 的速度设定（应与 1 在限幅/门控后一致）
ros2 topic echo /mavros/setpoint_velocity/cmd_vel_unstamped

# 3）飞控经 MAVROS 回读的 RC 通道输出（PWM 等；用于对比指令与最终通道是否饱和、延迟）
ros2 topic echo /mavros/rc/out
```

说明：**`/mavros/setpoint_velocity/cmd_vel_unstamped`** 由本项目 **`nav2_cmd_vel_to_mavros`**（或 relay）发布；**`/mavros/rc/out`** 为 MAVROS **订阅 MAVLink 后发布的 RC 输出反馈**（消息类型以 `ros2 topic info /mavros/rc/out` 为准），可与 **`/cmd_vel`**、setpoint 对照看 **时延、中断、饱和**。

## 相关话题备忘

| 话题 | 含义（概要） |
|------|----------------|
| `/mavros/local_position/odom` | **`nav_msgs/Odometry`**，模式 B 下作为 Nav2 里程输入（PX4 侧融合 GNSS + IMU 等，依飞控状态）。|
| `/mavros/home_position/home` | 飞控 **HOME**（及局地分量）回填；自检 **设 HOME** 是否生效。|
| **`/mavros/cmd/set_home`** | **服务** **`mavros_msgs/srv/CommandHome`**：将 PX4 HOME 设为当前船位（`current_gps: true`）或指定经纬高（见上节，`current_gps: false`）。|
| `/mavros/setpoint_velocity/cmd_vel_unstamped` | **`geometry_msgs/Twist`**：本项目速度桥写入的 OFFBOARD 速度设定（无 header）。联调：`ros2 topic echo ...`。 |
| `/mavros/rc/out` | MAVROS 发布的 **RC 输出反馈**（各通道 PWM 等，类型以 `ros2 topic info` 为准）；对照 **`/cmd_vel`** 与 setpoint，查混控输出是否跟上、是否饱和。 |
| `/mavros/manual_control/send` | 从 ROS **发往** MAVROS，再编码为 MAVLink 发给飞控。 |
| `/mavros/manual_control/control` | MAVROS **发布** 侧的手动量反馈/状态类信息，用于监视；**不能**当作“再发一轮指令”的输入口代替 `/send`。 |

差速船机上混控由 PX4 **机架/混控参数**决定；若 setpoint 有数据但推进异常，结合 **`/mavros/state`**、**`/mavros/rc/out`** 与解锁、模式继续排查。
