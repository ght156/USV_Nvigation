明白了，你现在看到那些 `/usv1/...` 节点消失了，是因为你把 ROS2 的网络发现隔离了，或者换了 `ROS_DOMAIN_ID`。我来详细解释一下原理和后续接实船时怎么做：

------

### 1️⃣ 为什么节点消失

- ROS2 基于 **DDS**（Data Distribution Service）通信，它会自动发现同一个 **ROS_DOMAIN_ID** 下局域网里的所有 ROS2 节点。
- 你之前看到的 `/usv1/...` 节点，很可能是局域网里另一台电脑、船载工控机或者 Docker 容器启动的 ROS2 节点。
- 当你执行：

```bash
export ROS_LOCALHOST_ONLY=1
```

或

```bash
export ROS_DOMAIN_ID=88
```

- DDS 会只发现本机的节点（localhost-only），或者只监听指定的 Domain ID。
- 所以那些 `/usv1` 节点就不再显示了，但本机自己的节点仍然正常运行。

------

### 2️⃣ 换 ROS_DOMAIN_ID 会影响你本地 ROS 项目吗？

- 不会，只要你的本地 ROS2 节点也使用同样的 `ROS_DOMAIN_ID`，它们之间通信就完全正常。
- 换了 `ROS_DOMAIN_ID` 后，你本机的 ROS2 节点和之前域里的其他 ROS2 节点 **就看不到对方** 了，隔离了网络。

------

### 3️⃣ 后面接实船或者需要通信怎么做

当你要和实船的 PX4、雷达、Livox 等通信时，你有两种方式：

#### **方案 A：统一 ROS_DOMAIN_ID**

- 确保船端和你本机的所有 ROS2 节点使用同一个 `ROS_DOMAIN_ID`（默认是 0）。
- 这样你的节点就可以直接看到船端节点，订阅 `/usv1/...` 的话题、发布控制命令。

#### **方案 B：保留不同 Domain 隔离，使用桥接节点**

- 如果船端和本机还有其他设备可能在同一个网络，但你不想被干扰，可以继续用不同 Domain ID。
- 然后通过 **ROS2 话题桥接** 或 **多 ROS_DOMAIN_ID 代理节点**（Bridge）进行必要的通信。
- 这种方式更安全，但需要额外配置桥接节点或 DDS Relay。

------

### 4️⃣ 建议

- **调试阶段**：用 `ROS_LOCALHOST_ONLY=1` 或不同 Domain ID，避免局域网其他 ROS2 节点干扰你的调试。
- **实船阶段**：统一 Domain ID 并关闭 `ROS_LOCALHOST_ONLY`，让你的本机节点可以看到船端 PX4、Livox 等节点。
- 记得设置网络可靠（最好直连局域网，避免 Wi-Fi 多播丢包）。

   - 确保 TF、话题命名与船端一致

### **静态 TF 配置**

假设你的 Livox 和船体安装位置固定，可以统一 TF：

```yaml
# static_transform_real_boat.yaml
transforms:
  - parent: "base_link"        # 船体基坐标系
    child: "livox_frame"       # Livox 坐标系
    translation:
      x: 1.2
      y: 0.0
      z: 0.0
    rotation:
      x: 0.0
      y: 0.0
      z: -0.0087
      w: 0.99996
```

> 注意 `parent` 和 `child` 与你 Nav2/船端 TF 保持一致
> **不要使用 `usv1/base_link`**，除非你全系统都用 namespace
